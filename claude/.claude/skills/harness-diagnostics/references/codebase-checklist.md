# Codebase 진단 체크리스트

> 코드베이스가 AI Agent에게 얼마나 친화적인지 평가하기 위한 종합 체크리스트.
> 총 84개 항목, 7개 카테고리.

---

## 1. Agent Entry Point (원칙 1) — 15항목

- [ ] 레포 루트에 `AGENTS.md` 또는 `CLAUDE.md` 파일이 존재하는가
- [ ] 프로젝트 한 줄 요약이 포함되어 있는가
- [ ] 기술 스택(언어, 프레임워크, 주요 라이브러리)이 명시되어 있는가
- [ ] 아키텍처 요약이 포함되어 있는가
- [ ] 빌드 명령어가 복사-붙여넣기 가능한 형태로 기재되어 있는가
- [ ] 실행 명령어가 기재되어 있는가
- [ ] 테스트 실행 명령어가 기재되어 있는가
- [ ] 단일 파일/모듈 테스트 명령어가 기재되어 있는가
- [ ] 디렉토리 구조 설명이 포함되어 있는가
- [ ] 코딩 컨벤션이 기재되어 있는가
- [ ] 흔한 실수/주의사항(gotchas)이 기재되어 있는가
- [ ] 관련 문서 링크가 제공되는가
- [ ] 환경 변수 및 설정 안내가 있는가
- [ ] 의존성 설치 방법이 명시되어 있는가
- [ ] 최근 코드 변경사항을 반영하고 있는가 (git log와 비교)

---

## 2. 문서 구조 (원칙 2, 5, 9) — 12항목

- [ ] `docs/` 디렉토리가 존재하는가
- [ ] `docs/` 내부가 카테고리별로 구조화되어 있는가
- [ ] ADR(Architecture Decision Records)이 존재하는가
- [ ] ADR에 결정 배경과 대안이 기록되어 있는가
- [ ] 개발 가이드가 문서화되어 있는가
- [ ] API 문서가 존재하는가 (OpenAPI, JSDoc 등)
- [ ] README.md가 최소한의 정보를 포함하는가
- [ ] Progressive Disclosure 구조 (요약 → 상세 → 링크)인가
- [ ] 문서 간 상호 참조(cross-reference)가 있는가
- [ ] 코드 내 인라인 문서가 "왜(why)"를 설명하는가
- [ ] CHANGELOG 또는 릴리스 노트가 관리되고 있는가
- [ ] 문서에 실행 가능한 코드 예제가 포함되어 있는가

---

## 3. Invariant 강제 (원칙 3, 4, 10) — 15항목

- [ ] Linter 설정 파일이 존재하는가 (ESLint, Pylint 등)
- [ ] Linter 규칙이 프로젝트에 적절한가
- [ ] Formatter 설정이 존재하는가 (Prettier, Black 등)
- [ ] Formatter가 자동 적용되는가 (저장 시 또는 commit 시)
- [ ] Type checking이 설정되어 있는가 (TypeScript strict, mypy 등)
- [ ] Type checking 엄격도가 적절한가 (strict mode, any 제한)
- [ ] CI/CD 파이프라인이 존재하는가
- [ ] CI에서 lint가 실행되는가
- [ ] CI에서 테스트가 실행되는가
- [ ] CI에서 type check가 실행되는가
- [ ] Pre-commit hooks가 설정되어 있는가
- [ ] 테스트 커버리지 기준이 설정되고 CI에서 강제되는가
- [ ] Branch protection 규칙이 설정되어 있는가
- [ ] Import 순서 규칙이 도구로 강제되는가
- [ ] 보안 검사(의존성 취약점 스캔)가 자동 실행되는가

---

## 4. 아키텍처 (원칙 6, 11) — 12항목

- [ ] 레이어 분리가 되어 있는가 (presentation/domain/infrastructure)
- [ ] 의존성 방향이 일관적인가 (순환 의존성 없음)
- [ ] 모듈 경계가 명확한가 (public API 정의)
- [ ] Import 규칙이 존재하고 강제되는가
- [ ] API contract가 명시적으로 정의되어 있는가
- [ ] 설정과 코드가 분리되어 있는가
- [ ] 공통 유틸리티가 중앙화되어 있는가
- [ ] 에러 처리 패턴이 프로젝트 전반에서 일관적인가
- [ ] 데이터 모델이 한 곳에서 정의되어 있는가
- [ ] 외부 의존성이 추상화 레이어를 통해 접근되는가
- [ ] 테스트 구조가 소스 구조와 대칭적인가
- [ ] 순환 의존성이 없는가 (도구로 검증)

---

## 5. Repository as Source of Truth (원칙 9, 12) — 10항목

- [ ] 기술적 결정이 레포 내(ADR, PR)에 기록되는가
- [ ] "왜 그렇게 구현했는지" 배경 정보가 코드/commit/PR에 있는가
- [ ] 계획/로드맵이 레포 내에서 관리되는가 (issues, ROADMAP.md)
- [ ] TODO/FIXME가 이슈와 연결되고 주기적으로 정리되는가
- [ ] 변수명, 함수명만으로 의도를 파악할 수 있는가
- [ ] 온보딩에 필요한 모든 정보가 레포에 있는가
- [ ] 외부 의존성 선택 이유가 기록되어 있는가
- [ ] 레포만으로 개발 환경을 처음부터 셋업할 수 있는가
- [ ] Commit 메시지가 변경의 의도를 충분히 설명하는가
- [ ] PR 템플릿이 존재하는가

---

## 6. 운영/유지보수 (원칙 7, 8) — 10항목

- [ ] Dead code 탐지 도구가 설정되어 있는가 (knip, ts-prune 등)
- [ ] Dead code가 주기적으로 제거되고 있는가
- [ ] 의존성이 주기적으로 업데이트되는가 (Dependabot, Renovate)
- [ ] Stale/deprecated 의존성이 식별되고 있는가
- [ ] 구조화된 로깅이 적용되어 있는가
- [ ] 에러 모니터링 도구가 연동되어 있는가 (Sentry, Datadog)
- [ ] Health check 엔드포인트가 있는가
- [ ] 불필요한 설정 파일이 정리되어 있는가
- [ ] Docker 이미지가 최적화되어 있는가
- [ ] Flaky test가 관리되고 있는가

---

## 7. Agent 가독성 (원칙 2, 5, 12) — 10항목

- [ ] 파일명과 디렉토리 구조만으로 코드 위치를 예측할 수 있는가
- [ ] 개별 파일이 적절한 크기를 유지하는가 (300줄 이하)
- [ ] 함수/메서드가 적절한 크기인가 (50줄 이하)
- [ ] 네이밍 컨벤션이 프로젝트 전반에서 일관적인가
- [ ] 애플리케이션 entry point가 명확히 식별 가능한가
- [ ] 모듈의 public API가 명확히 정의되어 있는가
- [ ] 매직 넘버/문자열이 상수로 추출되어 있는가
- [ ] 조건문/반복문 중첩이 과도하지 않은가 (3단계 이하)
- [ ] 같은 유형의 파일이 일관된 내부 구조를 따르는가
- [ ] 주석이 "what"이 아닌 "why"를 설명하는가
