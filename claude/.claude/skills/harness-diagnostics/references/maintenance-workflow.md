# Maintenance 모드 워크플로우

> 시간이 지난 프로젝트의 drift를 감지하고 정리 작업을 제안하기 위한 워크플로우.

---

## Phase 1: 변경 감지

Git 히스토리를 분석하여 최근 변경 사항을 파악한다.

### `git log` 분석

- 최근 변경 이력을 기간별로 분석
- 변경 빈도가 높은 파일 및 디렉토리 식별
- 커밋 메시지 패턴 분석

### 변경 파일 vs 문서 커버리지

- 코드가 변경되었으나 관련 문서가 업데이트되지 않은 영역 탐지
- AGENTS.md, CLAUDE.md의 최종 수정일 대비 코드 변경 이력 비교

### 새로운 의존성 추가

- `package.json`, `requirements.txt`, `go.mod` 등의 변경 감지
- 새로 추가된 의존성의 문서화 여부 확인

### 설정 변경

- CI/CD, linter, formatter 등 설정 파일 변경 감지
- 설정 변경이 문서에 반영되었는지 확인

## Phase 2: GC 대상 분류

정리가 필요한 항목을 유형별로 분류한다.

### Drift 유형

| 유형 | 설명 | 예시 |
|------|------|------|
| **문서 drift** | 현재 코드와 일치하지 않는 문서 | API 변경 후 미갱신 문서, 삭제된 기능 설명 잔존 |
| **Stale 코드** | 사용되지 않는 코드 요소 | 미사용 import, dead function, 주석 처리된 코드 |
| **설정 drift** | 실제 사용과 맞지 않는 설정 | 미사용 의존성, 비활성 CI 단계, 불일치 환경 변수 |
| **깨진 참조** | 유효하지 않은 링크 및 참조 | 문서 내 깨진 링크, 삭제된 파일 참조 |

### 심각도 분류

| 심각도 | 기준 |
|--------|------|
| **High** | 빌드 실패, 런타임 에러 유발 가능성 |
| **Medium** | 혼란 야기, 개발 효율 저하 |
| **Low** | 미관상 또는 정리 차원 |

### 노력 추정

- 각 항목의 수정에 필요한 예상 시간 산출
- 자동화 가능 여부 표시

---

## Phase 3: 델타 리포트

변경 사항과 drift 현황을 종합하여 리포트를 생성한다.

### 이전 audit 대비 변경 사항 (이전 기록이 있는 경우)

- 개선된 항목과 새로 발생한 문제 비교
- 이전 권장 사항의 이행 여부 확인

### 현재 drift 인벤토리

- 감지된 모든 drift 항목의 목록
- 유형, 심각도, 위치 정보 포함
- `bash scripts/maintenance-scan.sh` 결과를 기본 입력으로 사용

### GC 작업 목록

- 심각도와 노력 대비 효과를 고려한 우선순위 정렬
- 즉시 실행 가능한 항목 표시
- 자동화 가능한 항목 별도 분류
- orphaned references/examples/scripts, sync manifest 누락, stale self log 운영 항목을 별도 표시

### 성숙도 점수 추이 (이전 점수가 있는 경우)

- 시간에 따른 점수 변화 추적
- 개선 또는 퇴보 영역 식별
- 추세 기반 권장 사항

### 리포트 생성

- `references/report-template.md`의 Maintenance 리포트 형식 사용

### 리포트 형식 검증

리포트 생성 후 출력 전 다음 사항을 검증한다:

- [ ] 공통 헤더(진단 대상, 모드, 날짜, 기술 스택, 진단 범위)가 포함되어 있는가
- [ ] 변경 감지 요약이 수치(파일 수, 커밋 수 등)로 작성되어 있는가
- [ ] 모든 drift 항목에 유형, 위치, 심각도가 기재되어 있는가
- [ ] GC 작업 목록이 우선순위로 정렬되어 있는가
- [ ] 성숙도 추이에 이전/현재 점수와 변화 사유가 기재되어 있는가 (이전 기록이 있는 경우)
- [ ] `수행한 검증` 섹션에 실제 실행한 명령이 기록되어 있는가
- [ ] 자체 점검 부록이 포함되어 있는가
