# 성숙도 프레임워크 (Maturity Framework)

> 코드베이스의 에이전트 친화성을 평가하기 위한 5단계 성숙도 프레임워크.
> 12개 원칙을 4개 차원으로 그룹핑하고, 가중 평균으로 종합 점수를 산출한다.

---

## 1. 차원 그룹핑

### A. Documentation & Navigation — 가중치 30%

| 원칙 # | 원칙명 | 핵심 질문 |
|--------|--------|-----------|
| 1 | Agent Entry Point | 명확한 진입점이 있는가? |
| 2 | Map, Not Manual | 문서가 지도 역할을 하는가? |
| 5 | Progressive Disclosure | 정보가 계층적으로 구성되어 있는가? |
| 12 | Self-Documentation | 코드 자체가 의도를 설명하는가? |

**근거:** 에이전트가 코드베이스를 이해하고 올바른 위치를 찾는 것이 모든 작업의 전제 조건이다. 문서가 없으면 에이전트는 작업 자체를 시작할 수 없으므로, 가장 높은 가중치(30%)를 부여한다. 4개 원칙을 포함하는 이유는 진입점(P1), 문서 구조(P2), 정보 계층(P5), 코드 가독성(P12)이 모두 "에이전트가 올바른 위치에서 올바른 정보를 얻는가"라는 동일 관심사에 속하기 때문이다.

### B. Enforcement & Consistency — 가중치 30%

| 원칙 # | 원칙명 | 핵심 질문 |
|--------|--------|-----------|
| 3 | Invariant Enforcement | 실수를 도구가 자동으로 잡는가? |
| 4 | Convention Over Configuration | 명시적 규칙이 있는가? |
| 10 | Reproducibility | 동일 입력 → 동일 결과인가? |

**근거:** 일관성과 자동 검증은 에이전트의 출력 품질을 보장하는 핵심 메커니즘이다. 에이전트가 코드를 "찾는" 것(차원 A)만큼 "올바르게 작성하는" 것이 중요하므로, A와 동일한 30% 가중치를 부여한다. Invariant 강제(P3), 명시적 규칙(P4), 재현성(P10)은 모두 "에이전트가 실수 없이 일관된 결과를 내는가"라는 품질 보장 관심사를 공유한다.

### C. Architecture & Knowledge — 가중치 20%

| 원칙 # | 원칙명 | 핵심 질문 |
|--------|--------|-----------|
| 6 | Layered Architecture | 의존성 방향이 명확한가? |
| 9 | Knowledge in Repo | 지식이 레포에 있는가? |
| 11 | Modularity | 변경 영향 범위가 예측 가능한가? |

**근거:** 아키텍처와 도메인 지식은 에이전트의 작업 범위를 확장하는 기반이다. A/B 차원이 "기본적인 작업을 올바르게 수행하는 것"이라면, C 차원은 "복잡한 작업에서도 올바른 판단을 내리는 것"이다. 기본 동작 보장(A/B)보다는 중요도가 낮으므로 20% 가중치를 부여한다. 레이어 구조(P6), 도메인 지식(P9), 모듈성(P11)은 "에이전트가 변경의 영향 범위를 이해하는가"라는 관심사를 공유한다.

### D. Operations & Maintenance — 가중치 20%

| 원칙 # | 원칙명 | 핵심 질문 |
|--------|--------|-----------|
| 7 | Garbage Collection | stale 코드/문서를 정리하는가? |
| 8 | Observability | 에이전트가 결과를 검증할 수 있는가? |

**근거:** 운영 및 유지보수는 에이전트가 안정적으로 장기간 협업하기 위한 인프라 조건이다. 단기적 작업 품질(A/B)보다는 장기적 코드베이스 건강(drift 방지, 검증 가능성)에 해당하므로 20% 가중치를 부여한다. Garbage Collection(P7)과 Observability(P8)는 "시간이 지나도 에이전트가 효과적으로 작업할 수 있는가"라는 지속 가능성 관심사를 공유한다. 2개 원칙만 포함되지만, 가중치가 같은 이유는 장기적 영향력이 크기 때문이다.

## 2. 5등급 체계

| 등급 | 이름 | 종합 점수 | 특징 |
|------|------|----------|------|
| **L1** | **None** | 0–19 | 에이전트 협업 고려 없음 |
| **L2** | **Basic** | 20–39 | 최소한의 문서화 |
| **L3** | **Structured** | 40–59 | 체계적 구조, 부분 자동화 |
| **L4** | **Optimized** | 60–79 | 높은 자동화, 낮은 drift |
| **L5** | **Autonomous** | 80–100 | 에이전트가 독립적으로 작업 가능 |

## 3. 등급별 상세

### L1 — None (0–19점)

| 차원 | 상태 |
|------|------|
| A. Documentation | README 없거나 오래됨. AGENTS.md 부재. 탐색 진입점 없음. |
| B. Enforcement | linter/formatter 미설정. 코딩 패턴 파일마다 상이. CI 미구축. |
| C. Architecture | 모듈 경계 불명확, 순환 의존성. 도메인 용어 정의 없음. |
| D. Operations | 환경 설정 구두 전달. CHANGELOG 없음. |

### L2 — Basic (20–39점)

| 차원 | 상태 |
|------|------|
| A. Documentation | 기본 README 존재. 불완전하거나 최신 아님. |
| B. Enforcement | linter 설정 있으나 느슨. CI 있으나 커버리지 낮음. |
| C. Architecture | 대략적 모듈 경계. 일부 도메인 용어가 주석에 설명. |
| D. Operations | 기본 환경 설정 스크립트. 변경 이력은 git log만. |

### L3 — Structured (40–59점)

| 차원 | 상태 |
|------|------|
| A. Documentation | 구조화된 문서 체계. AGENTS.md 초기 버전 존재. |
| B. Enforcement | linter + formatter CI 강제. pre-commit hook 설정. |
| C. Architecture | 명시적 모듈 경계. 도메인 용어집 존재. |
| D. Operations | Docker로 환경 재현 가능. CHANGELOG 유지. |

### L4 — Optimized (60–79점)

| 차원 | 상태 |
|------|------|
| A. Documentation | 문서-코드 동기화 메커니즘. 포괄적 코드베이스 맵. |
| B. Enforcement | 엄격한 type-checking. 아키텍처 규칙 도구 검증. |
| C. Architecture | 모듈별 독립 빌드/테스트. DDD 수준 도메인 모델. |
| D. Operations | 원커맨드 환경 설정. 자동 릴리스. |

### L5 — Autonomous (80–100점)

| 차원 | 상태 |
|------|------|
| A. Documentation | Self-describing 코드베이스. 에이전트별 맞춤 안내 최적화. |
| B. Enforcement | Zero-tolerance. 패턴 준수율 95%+. 자동 마이그레이션 도구. |
| C. Architecture | 완전한 모듈 독립성. 실행 가능한 명세. 영향 자동 계산. |
| D. Operations | 완전 자동 프로비저닝. drift zero 수렴. |

## 4. 점수 계산

### 4.1 원칙 점수

각 원칙은 0–10 범위로 평가한다. (`references/principles.md` 참조)

### 4.2 차원 점수

각 차원 = 소속 원칙 점수의 산술 평균.

```
DimA = (P1 + P2 + P5 + P12) / 4
DimB = (P3 + P4 + P10) / 3
DimC = (P6 + P9 + P11) / 3
DimD = (P7 + P8) / 2
```

### 4.3 종합 점수

```
종합 = (DimA × 0.3 + DimB × 0.3 + DimC × 0.2 + DimD × 0.2) × 10
```

최대값: `(10×0.3 + 10×0.3 + 10×0.2 + 10×0.2) × 10 = 100`

## 5. 계산 예시

### 원칙별 점수

| 원칙 | 점수 | 근거 |
|------|------|------|
| P1 | 6 | README + 일부 ADR 존재, API 문서 미흡 |
| P2 | 5 | 루트에 디렉토리 설명 있으나 깊은 레벨 부재 |
| P3 | 7 | ESLint + Prettier CI 강제, type-check 부분적 |
| P4 | 6 | 주요 패턴 일관적, 레거시 코드 예외 다수 |
| P5 | 4 | AGENTS.md 없음, 일부 주석에 맥락 정보 |
| P6 | 7 | 명확한 패키지 분리, 일부 순환 의존성 |
| P7 | 8 | Docker Compose로 완전 재현 가능 |
| P8 | 5 | CHANGELOG 있으나 불규칙 업데이트 |
| P9 | 4 | 도메인 용어 정의 파일 부재 |
| P10 | 6 | pre-commit hook 있으나 커버리지 불완전 |
| P11 | 5 | package.json 관리 양호, 내부 의존성 불명확 |
| P12 | 5 | 기본 정보 접근 가능하나 깊은 맥락 부족 |

### 차원 점수

```
DimA = (6 + 5 + 4 + 5) / 4 = 5.0
DimB = (7 + 6 + 6) / 3 ≈ 6.33
DimC = (7 + 4 + 5) / 3 ≈ 5.33
DimD = (8 + 5) / 2 = 6.5
```

### 종합 점수

```
종합 = (5.0×0.3 + 6.33×0.3 + 5.33×0.2 + 6.5×0.2) × 10
     = (1.50 + 1.90 + 1.07 + 1.30) × 10
     = 5.77 × 10
     = 57.7 → L3 (Structured)
```

### 해석

- **종합 점수 57.7 → L3 (Structured)**
- 가장 약한 차원: A (5.0) — 에이전트 안내 문서 강화 필요
- 가장 강한 차원: D (6.5) — 환경 재현성이 높음

## 6. 등급 전환 기준

### L1 → L2

- [ ] 루트에 최신 README.md
- [ ] 하나 이상의 linter/formatter 설정
- [ ] 빌드/실행 방법 문서화
- [ ] 기본 CI 파이프라인

### L2 → L3

- [ ] AGENTS.md 또는 CLAUDE.md 작성
- [ ] linter + formatter CI 강제
- [ ] 주요 코딩 패턴 문서화
- [ ] pre-commit hook 설정
- [ ] 환경 설정 스크립트/컨테이너화

### L3 → L4

- [ ] 문서-코드 동기화 메커니즘
- [ ] 엄격한 type-checking 강제
- [ ] 아키텍처 규칙 도구 기반 검증
- [ ] 모듈별 독립 빌드/테스트 가능
- [ ] 원커맨드 환경 설정
- [ ] 모든 차원 점수 최소 5.0 이상

### L4 → L5

- [ ] 에이전트별 맞춤 안내 파일 최적화
- [ ] 모든 위반 자동 차단 (zero-tolerance)
- [ ] 패턴 준수율 95%+ (측정 가능)
- [ ] 완전한 모듈 독립성
- [ ] 완전 자동화 환경 프로비저닝
- [ ] 모든 차원 점수 최소 7.0 이상

## 7. 해석 가이드

| 등급 | 에이전트 활용 수준 |
|------|-------------------|
| L1 | 에이전트 활용 불가. 기본 인프라 구축 필요. |
| L2 | 단순 작업만 가능. 사람의 검토 비용 높음. |
| L3 | 정형화된 작업에 에이전트 활용 가능. 반복 업무 30-50% 절감. |
| L4 | 대부분의 일상적 작업을 독립 수행. 반복 업무 50-70% 절감. |
| L5 | 독립적 기여자 수준. 개발 속도 2-3배 향상. |

**주의**: 어떤 차원이든 3.0 미만이면, 종합 점수와 무관하게 해당 영역 에이전트 활용 불가.
