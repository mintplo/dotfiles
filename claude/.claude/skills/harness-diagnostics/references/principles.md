> OpenAI의 "Harness Engineering" 개념을 기반으로 구성.

## 1. Agent Entry Point

### 정의

AI 에이전트가 코드베이스에 처음 진입했을 때, 어디서부터 시작해야 하는지를 안내하는 명확한 진입점이 존재하는지를 평가한다.
`AGENTS.md`, `CLAUDE.md`, 또는 이에 준하는 파일이 레포 루트에 위치하여 에이전트에게 프로젝트 구조, 빌드 명령어, 테스트 방법, 핵심 규칙을 즉시 전달해야 한다.
이 진입점이 없으면 에이전트는 수십 개의 파일을 탐색하며 시간을 낭비하고, 잘못된 가정을 세울 가능성이 높아진다.

### 판단 기준 체크리스트

- [ ] 레포 루트에 `AGENTS.md`, `CLAUDE.md`, 또는 동등한 에이전트 진입점 파일이 존재하는가?
- [ ] 진입점 파일에 프로젝트의 목적, 기술 스택, 디렉토리 구조 개요가 포함되어 있는가?
- [ ] 빌드, 테스트, 린트 실행 명령어가 복사-붙여넣기 가능한 형태로 기술되어 있는가?
- [ ] 에이전트가 반드시 지켜야 할 핵심 규칙(금지 사항, 필수 패턴)이 명시되어 있는가?
- [ ] 하위 디렉토리별 `AGENTS.md`가 존재하거나, 진입점에서 하위 문서로의 링크가 제공되는가?

### 점수 기준

| 점수 | 기준 |
|------|------|
| **0** | 에이전트 진입점 파일이 전혀 없음. README.md도 에이전트 관점의 정보를 포함하지 않음. |
| **3** | README.md에 기본적인 빌드/테스트 명령어는 있으나, 에이전트 전용 안내가 없음. 디렉토리 구조 설명 부재. |
| **5** | `AGENTS.md` 또는 `CLAUDE.md`가 존재하지만, 내용이 불완전하거나 최신 상태가 아님. 핵심 규칙 일부 누락. |
| **8** | 진입점 파일이 존재하고, 프로젝트 구조/빌드/테스트/규칙을 모두 포함. 다만 하위 디렉토리별 세부 안내 부족. |
| **10** | 루트 진입점 + 주요 하위 디렉토리별 `AGENTS.md` 완비. 에이전트가 읽기만 하면 즉시 작업 가능. 정기적으로 갱신됨. |

### Anti-patterns

- **거대한 단일 README**: 수백 줄의 README.md에 인간용 마케팅 문구, 배지, 기여 가이드가 뒤섞여 에이전트가 필요한 정보를 추출하기 어려움.
- **구전 지식 의존**: "빌드는 그냥 `make`하면 돼"라고 팀원끼리만 아는 상태. 어디에도 문서화되지 않음.
- **Stale 진입점**: `AGENTS.md`가 존재하지만 6개월 전 상태. 현재 디렉토리 구조와 불일치.
- **숨겨진 진입점**: `.github/AGENT_GUIDE.md`처럼 에이전트가 발견하기 어려운 위치에 배치.
- **과도한 외부 링크 의존**: 진입점이 Confluence나 Notion 링크만 나열하여 에이전트가 접근 불가.

## 2. Map, Not Manual

### 정의

문서가 처음부터 끝까지 읽어야 이해되는 매뉴얼(manual)이 아니라, 필요한 정보를 즉시 찾아갈 수 있는 지도(map)의 역할을 하는지를 평가한다.
에이전트는 인간처럼 문서를 순차적으로 읽지 않는다. 특정 작업에 필요한 정보를 검색하고 점프하는 방식으로 소비한다.
따라서 문서는 목차, 크로스 레퍼런스, 명확한 섹션 구분을 통해 "현재 위치에서 원하는 곳으로 바로 갈 수 있는 지도"여야 한다.

### 판단 기준 체크리스트

- [ ] 주요 문서에 목차(Table of Contents)가 있고, 앵커 링크로 섹션 간 이동이 가능한가?
- [ ] 관련 문서 간 크로스 레퍼런스(상호 참조 링크)가 존재하는가?
- [ ] 각 문서가 단일 주제에 집중하고 있는가? (한 문서에 여러 주제가 혼합되지 않는가?)
- [ ] 디렉토리 구조 자체가 문서 탐색의 네비게이션 역할을 하는가?
- [ ] 검색 키워드로 원하는 정보를 빠르게 찾을 수 있도록 일관된 용어를 사용하는가?

### 점수 기준

| 점수 | 기준 |
|------|------|
| **0** | 문서가 거의 없거나, 존재하는 문서가 순차적 매뉴얼 형태. 목차 없음, 크로스 레퍼런스 없음. |
| **3** | 기본 문서는 존재하나, 긴 단일 파일에 모든 내용이 담겨 있음. 탐색이 어려움. |
| **5** | 주제별로 문서가 분리되어 있으나, 상호 참조가 부족하여 관련 문서를 찾기 어려움. |
| **8** | 목차, 크로스 레퍼런스, 주제별 분리가 잘 되어 있음. 일부 영역에서 탐색 경로 누락. |
| **10** | 완전한 지도 구조. 어떤 질문이든 2단계 이내의 탐색으로 답을 찾을 수 있음. 용어가 일관되고 검색 친화적. |

### Anti-patterns

- **소설형 문서**: "우리 프로젝트의 역사는 2019년으로 거슬러 올라갑니다..."로 시작하는 README.
- **단일 거대 문서**: `ARCHITECTURE.md`가 2000줄. 원하는 정보를 찾으려면 전체를 파싱해야 함.
- **고아 문서**: 어디에서도 링크되지 않는 문서. 존재를 알 수 없어 사실상 없는 것과 같음.
- **불일치 용어**: 같은 개념을 "서비스", "모듈", "컴포넌트"로 혼용하여 검색이 불가능.
- **순환 참조**: A 문서가 B를, B가 C를, C가 다시 A를 참조하여 실질적 정보에 도달 불가.

## 3. Invariant Enforcement

### 정의

코드베이스의 핵심 규칙(invariant)이 문서에만 적혀 있는 것이 아니라, 도구(linter, formatter, CI, pre-commit hook 등)에 의해 자동으로 강제되는지를 평가한다.
에이전트는 문서를 읽고 규칙을 "기억"하는 것보다, 규칙을 위반했을 때 즉각적인 피드백을 받는 환경에서 훨씬 더 정확하게 작업한다.
"문서에 쓰여 있으니 지켜라"는 접근은 실패한다. "위반하면 빌드가 깨진다"는 접근이 성공한다.

### 판단 기준 체크리스트

- [ ] 코드 스타일 규칙이 linter/formatter 설정 파일로 자동 강제되는가?
- [ ] CI 파이프라인에서 lint, type check, test가 필수로 실행되는가?
- [ ] pre-commit hook이 설정되어 커밋 시점에 기본 검증이 이루어지는가?
- [ ] 아키텍처 규칙(의존성 방향, import 제한 등)이 도구로 강제되는가?
- [ ] 규칙 위반 시 에러 메시지가 구체적이고, 수정 방법을 안내하는가?

### 점수 기준

| 점수 | 기준 |
|------|------|
| **0** | linter, formatter, CI가 전혀 없음. 모든 규칙이 구두 전달 또는 문서에만 존재. |
| **3** | 기본 linter/formatter는 있으나, CI에서 강제하지 않음. 로컬에서만 선택적으로 실행. |
| **5** | CI에서 lint + test가 실행되지만, 아키텍처 규칙이나 import 제한은 수동 리뷰에 의존. |
| **8** | lint, format, type check, test가 CI에서 필수. 주요 아키텍처 규칙도 도구로 강제. 에러 메시지가 명확. |
| **10** | 모든 핵심 invariant가 도구로 자동 강제. pre-commit hook 포함. 위반 시 구체적 수정 가이드 제공. |

### Anti-patterns

- **주석 규칙**: `// NOTE: 이 함수는 반드시 X 이후에 호출해야 합니다` - 도구가 아닌 주석으로만 규칙 전달.
- **선택적 CI**: CI가 있지만 실패해도 merge 가능. 규칙이 사실상 권고 사항.
- **비밀 규칙**: "우리 팀은 절대 any를 쓰지 않아"라고 하지만 tsconfig에 `strict: false`.
- **느린 피드백**: 규칙 위반을 30분 걸리는 CI 파이프라인 끝에서야 발견.
- **불명확한 에러**: `Error: lint failed` - 어디서 무엇이 왜 실패했는지 알 수 없는 메시지.

## 4. Convention Over Configuration

### 정의

프로젝트의 규칙과 패턴이 암묵적 합의가 아닌 명시적으로 문서화되고 코드에 반영되어 있는지를 평가한다.
에이전트는 팀의 암묵적 합의("우리는 보통 이렇게 해")를 알 수 없다. 파일 명명 규칙, 디렉토리 구조 패턴, API 설계 규칙 등이 명시적으로 기술되어야 한다.
기존 코드의 패턴을 따르는 것만으로도 올바른 코드를 작성할 수 있도록, 일관된 convention이 코드베이스 전반에 적용되어야 한다.

### 판단 기준 체크리스트

- [ ] 파일/디렉토리 명명 규칙이 문서화되어 있거나, 기존 구조에서 명확히 추론 가능한가?
- [ ] 새 기능/모듈 추가 시 따라야 할 템플릿 또는 예제가 존재하는가?
- [ ] API 엔드포인트, DB 스키마, 에러 코드 등의 명명 규칙이 일관적인가?
- [ ] 설정 파일이 최소화되어 있고, 대부분의 동작이 convention으로 결정되는가?
- [ ] convention을 어겼을 때 linter나 코드 리뷰에서 자동 감지할 수 있는가?

### 점수 기준

| 점수 | 기준 |
|------|------|
| **0** | 규칙 없음. 같은 프로젝트 내에서 camelCase, snake_case, kebab-case가 혼재. 패턴 추론 불가. |
| **3** | 일부 영역에서 일관성 있으나, 프로젝트 전반의 통일된 convention 부재. 문서화되지 않음. |
| **5** | convention이 존재하고 대부분 따르고 있으나, 명시적 문서가 없어 기존 코드에서 역추론해야 함. |
| **8** | convention이 문서화되어 있고 코드베이스 전반에서 일관적. 새 코드 추가 시 패턴이 명확. |
| **10** | convention이 문서화 + 도구로 강제 + 코드 생성기/템플릿 제공. 에이전트가 기존 패턴을 복제하기만 하면 올바른 코드 생성 가능. |

### Anti-patterns

- **혼합 스타일**: 같은 디렉토리에 `userService.ts`, `product_handler.ts`, `OrderRepo.ts`가 공존.
- **마법의 문자열**: 코드 곳곳에 하드코딩된 문자열이 convention 대신 사용됨.
- **과도한 설정**: 수백 줄의 설정 파일. convention 대신 설정이 모든 동작을 결정.
- **암묵적 순서 의존**: "이 파일들은 이 순서로 import 해야 해" - 코드에 반영되지 않은 암묵적 규칙.
- **예외의 일반화**: convention을 정했지만 예외가 너무 많아 사실상 규칙이 없는 것과 같음.

## 5. Progressive Disclosure

### 정의

정보가 한꺼번에 쏟아지는 것이 아니라, 에이전트가 작업을 진행하면서 필요한 시점에 필요한 깊이의 정보를 발견할 수 있도록 계층적으로 구성되어 있는지를 평가한다.
진입점에서는 전체 구조를 빠르게 파악하고, 특정 모듈을 작업할 때는 해당 모듈의 상세 문서를 읽고, 엣지 케이스를 처리할 때는 관련 ADR을 참조하는 식이다.

### 판단 기준 체크리스트

- [ ] 정보가 최소 3단계(개요 → 모듈별 상세 → 엣지 케이스/의사결정 기록)로 계층화되어 있는가?
- [ ] 각 디렉토리에 해당 범위의 문서(README, AGENTS.md 등)가 존재하는가?
- [ ] 코드 내 주석이 "왜(why)"에 집중하고, "무엇(what)"은 코드 자체에 위임하는가?
- [ ] ADR(Architecture Decision Record) 또는 이에 준하는 의사결정 기록이 존재하는가?
- [ ] 에이전트가 현재 작업 범위에서 불필요한 정보에 노출되지 않는가?

### 점수 기준

| 점수 | 기준 |
|------|------|
| **0** | 정보 계층이 없음. 모든 것이 루트 README 하나 또는 코드 주석에만 존재. |
| **3** | 루트 문서와 일부 모듈 문서가 있으나, 깊이 간 연결이 부족. |
| **5** | 개요 → 상세의 2단계 구조. ADR이나 의사결정 기록은 없음. |
| **8** | 3단계 이상의 정보 계층. 모듈별 문서 + ADR 존재. 대부분의 작업에서 필요한 정보를 적시에 발견 가능. |
| **10** | 완전한 progressive disclosure. 어떤 깊이의 작업이든 해당 깊이에서 필요한 정보가 과부족 없이 존재. |

### Anti-patterns

- **정보 폭포**: 루트 README가 500줄 이상. 에이전트가 모든 정보를 초기에 소비해야 함.
- **바닥 없는 코드**: 모듈 내부에 들어갔을 때 아무런 문서도 없음. 코드만 존재하고 맥락 부재.
- **평면적 구조**: 모든 문서가 `/docs` 한 폴더에 평면적으로 나열. 깊이 개념 없음.
- **의사결정 블랙홀**: "왜 이 기술을 선택했는지"에 대한 기록이 없음. Slack 스레드에만 존재.

## 6. Layered Architecture

### 정의

코드베이스의 의존성 방향이 명확하게 정의되어 있어, 에이전트가 한 레이어를 수정할 때 영향 범위를 예측할 수 있는지를 평가한다.
상위 레이어가 하위 레이어에 의존하되, 반대 방향의 의존은 금지되는 등 명확한 규칙이 필요하다.

### 판단 기준 체크리스트

- [ ] 레이어 간 의존성 방향이 문서화되어 있는가?
- [ ] 순환 의존성(circular dependency)이 없는가? 또는 도구로 감지되는가?
- [ ] 각 레이어의 책임과 경계가 디렉토리 구조에 반영되어 있는가?
- [ ] 레이어 간 통신이 명확한 인터페이스를 통해 이루어지는가?
- [ ] 의존성 위반을 자동으로 감지하는 도구가 설정되어 있는가?

### 점수 기준

| 점수 | 기준 |
|------|------|
| **0** | 레이어 개념 없음. 모든 코드가 단일 디렉토리에 혼재. 의존성 방향 무질서. |
| **3** | 디렉토리 분리는 있으나, 레이어 간 경계가 불분명. 순환 의존성 존재. |
| **5** | 레이어가 구분되고 대부분의 의존성 방향이 올바르나, 일부 위반 존재. 도구 강제 없음. |
| **8** | 명확한 레이어 구조 + 문서화 + 대부분의 의존성 규칙 준수. 일부 도구 강제. |
| **10** | 완전한 레이어 아키텍처. 의존성 방향 도구로 강제. 순환 의존성 zero. |

### Anti-patterns

- **God 모듈**: 모든 것을 아는 중앙 모듈. 모든 레이어가 이것에 의존.
- **레이어 관통**: presentation 레이어에서 DB를 직접 호출.
- **양방향 의존**: A 모듈이 B를, B 모듈이 A를 import. 순환 참조.
- **공유 유틸의 비대화**: `utils/`, `common/`이 비대해져 사실상 암묵적 God 모듈.

## 7. Garbage Collection

### 정의

더 이상 사용되지 않는 코드, 오래된 문서, deprecated된 API, 죽은 feature flag 등을 주기적으로 정리하여 코드베이스를 깨끗하게 유지하는지를 평가한다.
에이전트는 레포에 있는 모든 코드를 "유효한 것"으로 간주한다. dead code가 많으면 에이전트가 잘못된 패턴을 참조하거나, 이미 삭제된 API를 호출할 수 있다.

### 판단 기준 체크리스트

- [ ] dead code 탐지 도구가 CI에 통합되어 있는가?
- [ ] deprecated된 코드/API에 `@deprecated` 태그와 대안이 명시되어 있는가?
- [ ] 사용되지 않는 feature flag가 주기적으로 정리되는가?
- [ ] TODO/FIXME/HACK 주석이 방치되지 않고 관리되는가?
- [ ] 문서가 코드 변경에 맞춰 함께 갱신되는가?

### 점수 기준

| 점수 | 기준 |
|------|------|
| **0** | dead code, stale 문서가 대량으로 존재. 어떤 코드가 실제로 사용되는지 판단 불가. |
| **3** | deprecated 표시는 일부 있으나, 실제 정리는 이루어지지 않음. |
| **5** | 주기적 정리가 가끔 이루어짐. `@deprecated` 사용. 자동화된 탐지 없음. |
| **8** | dead code 탐지 도구 사용 + 주기적 정리 프로세스 존재. TODO에 이슈 링크. |
| **10** | 자동화된 garbage collection. dead code가 CI에서 경고. feature flag에 만료일. |

### Anti-patterns

- **코드 묘지**: 삭제 대신 주석 처리된 코드 블록이 수십 개.
- **영원한 TODO**: `// TODO: 나중에 리팩토링` - 2년 전에 작성된 TODO.
- **좀비 feature flag**: 이미 100% 롤아웃된 feature flag가 코드에 남아 있음.
- **시간 캡슐 문서**: "현재 v2를 개발 중입니다" - 이미 v5인데 v2 시절 문서가 남아 있음.

## 8. Observability

### 정의

에이전트가 자신이 수행한 작업의 결과를 스스로 검증할 수 있는 피드백 메커니즘이 존재하는지를 평가한다.
코드를 작성한 후 테스트를 실행하고, 빌드를 확인하고, lint를 통과하는 것 - 이 모든 것이 에이전트에게 열려 있어야 한다.

### 판단 기준 체크리스트

- [ ] 단위 테스트를 특정 파일/모듈 단위로 빠르게 실행할 수 있는가?
- [ ] 테스트 실행 명령어가 간단하고 문서화되어 있는가?
- [ ] type check 명령어가 존재하고, 빠르게 실행 가능한가?
- [ ] 빌드 실패 시 에러 메시지가 구체적이고 원인을 특정할 수 있는가?
- [ ] 로컬 환경에서 변경 결과를 즉시 확인할 수 있는 방법이 있는가?

### 점수 기준

| 점수 | 기준 |
|------|------|
| **0** | 테스트 없음. 빌드 확인 방법 불명. 에이전트가 코드의 정확성을 검증할 수단 전무. |
| **3** | 테스트가 존재하나 실행이 복잡하거나 느림 (10분 이상). 부분 실행 불가. |
| **5** | 테스트 + type check 실행 가능. 다만 특정 모듈 단위의 빠른 검증은 어려움. |
| **8** | 모듈 단위 테스트 + type check + lint를 빠르게 실행 가능. 에러 메시지 명확. |
| **10** | 수초 내 피드백 가능. 파일 단위 테스트, 증분 빌드, 명확한 에러 메시지. |

### Anti-patterns

- **전체 아니면 무**: 단일 파일 테스트가 불가능. 전체 테스트 스위트(30분)만 실행 가능.
- **침묵의 실패**: 테스트가 통과하지만 assertion이 없는 빈 테스트.
- **환경 의존 테스트**: 특정 DB, 외부 서비스가 있어야만 테스트 실행 가능.
- **비결정적 테스트**: 실행할 때마다 결과가 달라지는 flaky test.

## 9. Knowledge in Repo

### 정의

프로젝트를 이해하고 작업하는 데 필요한 지식이 팀원의 머릿속이나 외부 도구(Slack, Notion)가 아닌 레포지토리 자체에 존재하는지를 평가한다.
에이전트는 Slack 히스토리를 검색하거나, 시니어 개발자에게 질문할 수 없다. 레포에 없는 지식은 에이전트에게 존재하지 않는 것이다.

### 판단 기준 체크리스트

- [ ] 주요 아키텍처 결정의 이유가 ADR 또는 코드 내 주석으로 기록되어 있는가?
- [ ] 비즈니스 규칙이 코드에 반영되어 있고, 해당 규칙의 출처/근거가 문서에 있는가?
- [ ] 환경 설정, 배포 절차, 트러블슈팅 가이드가 레포 내 문서에 있는가?
- [ ] 외부 서비스 연동의 제약 조건, API 특이사항이 코드 근처에 문서화되어 있는가?
- [ ] 온보딩에 필요한 모든 정보가 레포만으로 충족 가능한가?

### 점수 기준

| 점수 | 기준 |
|------|------|
| **0** | 대부분의 지식이 특정 팀원의 머릿속에만 존재. "그건 민수에게 물어봐." |
| **3** | 기본적인 설정 문서는 레포에 있으나, 아키텍처 결정, 비즈니스 규칙의 근거는 외부에만 존재. |
| **5** | 주요 문서가 레포에 있으나, 일부 중요한 맥락이 외부에만 존재. ADR 없음. |
| **8** | ADR 존재, 비즈니스 규칙 문서화, 트러블슈팅 가이드 포함. 일부 암묵적 지식만 누락. |
| **10** | 레포가 single source of truth. 외부 문서 참조 불필요. |

### Anti-patterns

- **구전 전통**: 코드 어디에도 이유가 기록되지 않은 금기 사항.
- **Slack 아카이브**: 핵심 의사결정이 Slack 스레드에만 존재. 3개월 후 검색 불가.
- **Notion 의존**: "자세한 건 Notion 페이지 참고" - 에이전트가 접근할 수 없음.
- **부족 지식**: 특정 모듈은 오직 한 사람만 이해.

## 10. Reproducibility

### 정의

동일한 입력(코드, 설정, 환경)이 주어졌을 때 항상 동일한 결과(빌드, 테스트, 배포)가 나오는지를 평가한다.
에이전트가 어제 성공한 것과 동일한 작업을 오늘 실행했을 때 실패한다면, 에이전트는 자신의 변경이 문제인지 환경이 문제인지 구분할 수 없다.

### 판단 기준 체크리스트

- [ ] 의존성 버전이 lock 파일로 고정되어 있는가?
- [ ] 빌드가 결정론적인가? 동일 커밋에서 동일 결과물이 생성되는가?
- [ ] 개발 환경 설정이 자동화되어 있는가? (Docker, devcontainer, Nix 등)
- [ ] 테스트가 외부 상태(네트워크, DB, 시간)에 의존하지 않는가?
- [ ] CI 환경과 로컬 환경의 차이가 최소화되어 있는가?

### 점수 기준

| 점수 | 기준 |
|------|------|
| **0** | lock 파일 없음. "내 컴퓨터에서는 되는데?" 빈발. 환경 설정 문서 없음. |
| **3** | lock 파일은 있으나, 환경 설정이 수동. CI와 로컬 환경 차이 큼. |
| **5** | 의존성 고정 + 기본 환경 설정 자동화. 일부 테스트가 외부 상태에 의존. |
| **8** | 결정론적 빌드 + 컨테이너화된 환경 + 대부분의 테스트가 격리됨. |
| **10** | 완전한 재현성. 어떤 머신에서든 동일 결과. 모든 테스트가 격리. |

### Anti-patterns

- **부유하는 의존성**: `"react": "^18.0.0"` - 설치 시점에 따라 다른 버전이 설치됨.
- **snowflake 서버**: CI 서버에 수동으로 설치된 도구에 의존.
- **시간 의존 테스트**: `Date.now()`를 사용하여 자정 전후로 실패하는 테스트.
- **암묵적 전역 상태**: 테스트 간 DB 상태가 공유되어 실행 순서에 따라 결과가 달라짐.

## 11. Modularity

### 정의

코드베이스가 독립적으로 이해하고 수정할 수 있는 모듈로 구성되어 있어, 한 곳의 변경이 예측 가능한 범위에만 영향을 미치는지를 평가한다.

### 판단 기준 체크리스트

- [ ] 각 모듈이 명확한 public API를 가지고, 내부 구현이 캡슐화되어 있는가?
- [ ] 모듈 간 의존성이 최소화되어 있는가? (낮은 결합도)
- [ ] 한 모듈 내부의 코드가 높은 응집도를 보이는가?
- [ ] 모듈을 독립적으로 테스트할 수 있는가?
- [ ] 새 기능을 추가할 때 기존 모듈을 수정하지 않고 새 모듈을 추가하는 것이 자연스러운가?

### 점수 기준

| 점수 | 기준 |
|------|------|
| **0** | 모듈 개념 없음. 수천 줄의 단일 파일. 모든 것이 모든 것에 의존. |
| **3** | 파일은 분리되어 있으나, 모듈 경계 불명확. 한 파일 수정 시 10개 이상의 파일에 영향. |
| **5** | 모듈 구조 존재. 대부분의 모듈이 독립적이나, 일부 강결합 영역 존재. |
| **8** | 명확한 모듈 경계 + public API 정의 + 독립 테스트 가능. |
| **10** | 완전한 모듈화. 각 모듈이 독립적으로 이해/수정/테스트 가능. 변경 영향 범위가 항상 예측 가능. |

### Anti-patterns

- **God 클래스/파일**: 3000줄짜리 `UserService.ts`가 인증, 프로필, 결제, 알림을 모두 처리.
- **전역 상태 남용**: 모듈이 전역 변수/싱글톤을 통해 암묵적으로 통신.
- **Shotgun surgery**: 하나의 기능 변경을 위해 10개 이상의 파일을 동시에 수정해야 함.
- **깊은 상속 체인**: 부모 변경 시 모든 자식에 영향.

## 12. Self-Documentation

### 정의

코드 자체가 별도의 문서 없이도 자신의 의도와 동작을 설명하는지를 평가한다.
변수명, 함수명, 타입 정의, 디렉토리 구조가 충분히 서술적이어서 에이전트가 코드를 읽는 것만으로 무엇을 하는지, 왜 그렇게 하는지를 이해할 수 있어야 한다.

### 판단 기준 체크리스트

- [ ] 함수/변수/클래스 이름이 역할과 의도를 충분히 전달하는가?
- [ ] 타입 시스템이 적극적으로 활용되어 가능한 값의 범위를 코드로 표현하는가?
- [ ] 매직 넘버, 매직 스트링이 이름 있는 상수로 대체되어 있는가?
- [ ] 복잡한 비즈니스 로직에 "왜(why)" 주석이 있는가?
- [ ] 함수의 크기가 적절하여 (20-30줄 이내), 함수명만으로 동작을 예측할 수 있는가?

### 점수 기준

| 점수 | 기준 |
|------|------|
| **0** | `a`, `tmp`, `data`, `handleClick2` 수준의 이름. 타입 없음. 매직 넘버 다수. |
| **3** | 이름이 어느 정도 서술적이나, 불일치가 많음. 타입이 부분적. |
| **5** | 대부분의 이름이 서술적. 타입 시스템 활용. 다만 복잡한 로직에 "왜" 주석 부족. |
| **8** | 서술적 이름 + 강한 타입 + named constant + "왜" 주석. 대부분의 코드를 읽기만 해도 이해 가능. |
| **10** | 코드 자체가 완전한 문서. 이름, 타입, 구조, 주석이 완벽히 조화. |

### Anti-patterns

- **약어 남용**: `usrSvcMgr`, `calcTtlAmt`. 해독이 필요한 이름.
- **거짓말하는 이름**: `isValid`라는 함수가 실제로는 데이터를 변환함.
- **주석이 코드를 반복**: `// 사용자를 가져온다` + `getUser()`.
- **any 지옥**: TypeScript를 사용하지만 `any`가 수십 곳.
- **500줄 함수**: 이름으로 동작을 요약하는 것이 불가능한 크기.

## 우선순위 권장 순서

개선 효과가 가장 큰 순서 (ROI 기준):

1. **Invariant Enforcement** (3) - 도구로 자동 강제하면 에이전트 오류율이 극적으로 감소
2. **Agent Entry Point** (1) - 진입점 하나로 에이전트의 초기 탐색 시간을 90% 단축
3. **Observability** (8) - 에이전트가 자체 검증할 수 있으면 결과 품질 보장
4. **Convention Over Configuration** (4) - 일관된 패턴은 에이전트의 패턴 매칭 능력 극대화
5. **Knowledge in Repo** (9) - 레포에 지식이 있어야 에이전트가 접근 가능
6. **Reproducibility** (10) - 동일 결과 보장으로 에이전트의 시행착오 감소
7. **Modularity** (11) - 변경 범위 예측으로 안전한 수정 가능
8. **Layered Architecture** (6) - 의존성 방향 이해로 부작용 방지
9. **Self-Documentation** (12) - 코드 가독성이 에이전트의 이해도 직결
10. **Map, Not Manual** (2) - 탐색 효율 개선
11. **Progressive Disclosure** (5) - 정보 과부하 방지
12. **Garbage Collection** (7) - 거짓 신호 제거

## 정량 판정 보조 지표

점수 부여 시 주관을 최소화하기 위한 정량 기준. 판단 기준 체크리스트의 충족 수와 병행하여 사용한다.

| 원칙 | 정량 지표 | 0점 | 3점 | 5점 | 8점 | 10점 |
|------|-----------|-----|-----|-----|-----|------|
| P1. Agent Entry Point | 체크리스트 5항목 충족 수 | 0개 | 1개 | 2-3개 | 4개 | 5개 |
| P2. Map, Not Manual | 문서 간 상호 참조 비율 | 0% | < 30% | 30-60% | 60-90% | > 90% |
| P3. Invariant Enforcement | CI 자동 실행 검증 항목 수 (lint/type/test/arch) | 0개 | 1개 | 2개 | 3개 | 4개 + pre-commit |
| P4. Convention Over Configuration | 파일명 규칙 준수율 | < 50% | 50-70% | 70-85% | 85-95% | > 95% |
| P5. Progressive Disclosure | 정보 계층 단계 수 | 0-1단계 | 1단계 | 2단계 | 3단계 | 3단계 + 각 계층 완비 |
| P6. Layered Architecture | 순환 의존성 수 | 측정 불가 | 5개+ | 2-4개 | 1개 | 0개 + 도구 강제 |
| P7. Garbage Collection | dead code / stale 문서 비율 | > 20% | 10-20% | 5-10% | 2-5% | < 2% + 자동 탐지 |
| P8. Observability | 피드백 루프 소요 시간 | 없음 | > 10분 | 3-10분 | 1-3분 | < 1분 |
| P9. Knowledge in Repo | 외부 전용 지식 비율 | > 80% | 50-80% | 20-50% | 5-20% | < 5% |
| P10. Reproducibility | "내 컴퓨터에서는 되는데" 빈도 | 매주 | 매달 | 분기 1회 | 반기 1회 | 0회 |
| P11. Modularity | 1개 변경 시 영향 받는 평균 파일 수 | > 10개 | 6-10개 | 3-5개 | 2개 | 1개 |
| P12. Self-Documentation | 의도 파악에 외부 문서 참조 필요한 함수 비율 | > 50% | 30-50% | 15-30% | 5-15% | < 5% |

**사용법**: 체크리스트 기반 정성 평가와 정량 지표를 교차 확인하여 최종 점수를 결정한다. 두 기준의 점수 차가 2점 이상이면 근거를 재검토한다.

## 정량 지표 측정 절차

각 지표의 구체적 측정 방법. 측정 시 도구나 명령어를 명시하여 평가자 간 재현성을 보장한다.

| 원칙 | 지표 | 측정 절차 |
|------|------|-----------|
| P1 | 체크리스트 충족 수 | `references/principles.md` P1 판단 기준 체크리스트 5항목을 순서대로 확인. 충족 = Y, 미충족 = N으로 기록 후 Y 개수를 합산. |
| P2 | 상호 참조 비율 | 전체 문서 파일 수(N)를 센다. 그 중 다른 문서로의 링크/참조를 1개 이상 포함하는 파일 수(M)를 센다. 비율 = M/N × 100. |
| P3 | CI 검증 항목 수 | CI 설정 파일(`.github/workflows/`, `.gitlab-ci.yml` 등)을 읽고, lint / type-check / test / architecture-check 4가지 중 실행되는 항목을 센다. pre-commit은 `.husky/`, `.pre-commit-config.yaml` 존재 여부로 판정. |
| P4 | 파일명 준수율 | `src/` (또는 주요 소스 디렉토리)의 파일 목록을 추출. 프로젝트 convention에 맞는 파일명 수(M) / 전체 파일 수(N) × 100. convention이 미정의 시 가장 빈번한 패턴을 기준으로 삼는다. |
| P5 | 정보 계층 수 | 루트 문서(AGENTS.md/README) = 1단계. 하위 디렉토리별 문서(모듈 README) = 2단계. ADR/상세 설계 문서 = 3단계. 각 단계의 존재 여부를 확인. |
| P6 | 순환 의존성 수 | 언어별 도구 사용: `madge --circular` (JS/TS), `pydeps --show-cycles` (Python), `go vet` (Go). 도구 미사용 시 주요 모듈 10개의 import를 수동 추적하여 순환 여부 확인. |
| P7 | stale 비율 | dead code: 언어별 도구(`knip`, `ts-prune`, `vulture`)로 탐지. stale 문서: 최종 수정일이 관련 소스 코드 최종 수정일보다 3개월+ 오래된 문서 수 / 전체 문서 수 × 100. |
| P8 | 피드백 소요 시간 | 단일 파일에 1줄 변경 후 `테스트 실행` ~ `결과 확인`까지의 벽시계 시간을 측정. 3회 측정 후 중앙값 사용. |
| P9 | 외부 전용 지식 비율 | "이 정보는 레포 내에서 찾을 수 있는가?"를 기준으로 주요 지식 항목 10개(아키텍처 결정, 배포 절차, 비즈니스 규칙 등)를 나열. 레포에 없는 항목 수 / 10 × 100. |
| P10 | 환경 재현 실패 빈도 | 최근 3개월간 CI 실패 또는 "로컬에서만 성공" 이슈를 git log / issue tracker에서 검색. 횟수를 기간으로 환산. |
| P11 | 변경 영향 파일 수 | 최근 PR/커밋 10개를 샘플링. 각 PR에서 변경된 파일 수의 평균을 산출. 단, 대규모 리팩토링은 제외. |
| P12 | 외부 문서 필요 함수 비율 | 주요 모듈 3개에서 public 함수 20개를 샘플링. 함수명 + 시그니처 + 인접 주석만으로 의도를 파악할 수 있는지 판정. 파악 불가 함수 수 / 20 × 100. |
