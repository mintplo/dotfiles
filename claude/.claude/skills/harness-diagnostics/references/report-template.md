# 리포트 템플릿

> harness-diagnostics skill의 3가지 모드(Setup, Audit, Maintenance)에 대한 리포트 템플릿.
> 각 리포트는 공통 헤더로 시작하며, 모드별 본문과 자체 점검 부록으로 구성된다.

---

## 공통 헤더

모든 리포트의 최상단에 포함:

```markdown
# 하니스 진단 리포트

- **진단 대상**: [프로젝트명 / Skill명]
- **모드**: [Setup / Audit / Maintenance]
- **날짜**: YYYY-MM-DD
- **기술 스택**: [detected tech stack]
- **진단 범위**: [코드베이스 / Skill / Self]
```

### 공통 필수 섹션

모든 리포트 본문에는 아래 섹션이 반드시 포함되어야 한다.

- `요약`
- `근거`
- `위험 요소`
- `수행한 검증`

한두 문단으로 현재 상태, 가장 큰 gap, 권장 첫 작업을 요약한다.

### 2. 요구사항 오버레이

사용자가 별도로 제공한 요구사항이 있을 경우, 현재 상태와 분리해서 기록한다.

| 항목 | 내용 | 분류 |
|------|------|------|
| 제품/서비스 목적 | | 사용자 요구사항 |
| 기술 스택 제약 | | 사용자 요구사항 |
| 팀 운영 규칙 | | 사용자 요구사항 |
| 품질 기준 | | 사용자 요구사항 |
| 문서 요구사항 | | 사용자 요구사항 |
| 테스트/CI 요구사항 | | 사용자 요구사항 |

추가로, Setup 리포트의 주요 판단과 제안에는 아래 출처 라벨을 함께 사용한다.

- `레포 근거`: 레포에서 직접 확인한 사실
- `사용자 요구사항`: 사용자가 입력으로 제공한 목표/제약
- `추론`: 레포 근거와 요구사항을 바탕으로 도출한 제안 또는 추론

예시 형식:

```markdown
- **레포 근거**: `package.json`, `tsconfig.json`, `.github/workflows/ci.yml` 미존재
- **사용자 요구사항**: "TypeScript strict, PR 필수, CI 필요"
- **추론**: AGENTS.md + strict tsconfig + 최소 CI를 Immediate 우선순위로 제안
```

## 자체 점검 부록

모든 리포트 마지막에 포함:

```markdown
## 부록: 자체 점검

이 진단 리포트 자체의 품질:
- [ ] 모든 원칙에 대해 근거가 제시되었는가?
- [ ] 점수가 일관된 기준으로 부여되었는가?
- [ ] 제안 사항이 구체적이고 실행 가능한가?
- [ ] 우선순위가 명확한가?
- [ ] 리포트가 읽기 쉬운가?
```

---

## 리포트 작성 원칙

1. **근거 기반**: 모든 점수/판단에 구체적 근거(파일 경로, 코드 조각) 제시
2. **실행 가능성**: "무엇을 어떻게 해야 하는지" 명확히
3. **우선순위 명시**: 모든 작업 항목에 우선순위 부여
4. **일관성**: 동일 기준으로 시간 경과에 따른 비교 가능
5. **자체 점검 필수**: 리포트 마지막에 반드시 포함
6. **출처 분리**: Setup 모드에서는 `레포 근거`, `사용자 요구사항`, `추론`을 혼용하지 않음
