# Skill 진단 체크리스트

> Codex skill의 품질을 체계적으로 평가하기 위한 체크리스트.
> 이 체크리스트는 자기 자신(harness-diagnostics skill)에게도 동일하게 적용된다.

---

## 1. 파일 구조 검증 — 7항목

- [ ] `SKILL.md`가 skill 루트 디렉토리에 존재하는가
- [ ] `SKILL.md`가 유일한 진입점(entry point)으로 기능하는가
- [ ] 레포 운영용 `AGENTS.md`가 존재하는가
- [ ] `references/` 디렉토리가 존재하고, 상세 참조 문서를 포함하는가
- [ ] `examples/` 디렉토리가 존재하는가 (필요한 경우)
- [ ] 파일명이 kebab-case 규칙을 따르는가 (예: `skill-checklist.md`)
- [ ] 불필요한 파일(임시 파일, 중복 문서, 빈 파일)이 없는가
- [ ] 디렉토리 깊이가 2단계 이내인가 (`skill/references/file.md`)

---

## 2. SKILL.md 품질 — 6항목

- [ ] 지도(Map) 역할을 하는가 — 매뉴얼이 아닌, 어디에 무엇이 있는지 안내
- [ ] 전체 길이가 약 120줄 이내로 간결한가
- [ ] Progressive Disclosure를 따르는가 — 개요만 제공하고 상세는 references로 위임
- [ ] 정보 중복이 없는가 — SKILL.md와 references 간 같은 내용을 반복하지 않음
- [ ] 명확한 섹션 구분이 있는가 (원칙, 모드, 대상, 워크플로우, 참조 테이블)
- [ ] frontmatter가 존재하고 올바른 형식인가 (YAML `---` 블록)

---

## 3. Frontmatter 품질 — 6항목

- [ ] `name` 필드가 kebab-case로 작성되어 있는가
- [ ] `description` 필드가 존재하고, 목적과 트리거 조건을 명시하는가
- [ ] 트리거 조건이 구체적인가 — 어떤 상황에서 활성화되는지 명확
- [ ] 트리거 표현 예시가 포함되어 있는가 (사용자가 쓸 법한 표현)
- [ ] `version` 필드가 존재하고 semver 형식인가
- [ ] 불필요한 메타데이터 필드가 없는가

---

## 4. 본문 구조 — 6항목

- [ ] **핵심 원칙** 섹션이 존재하는가 — 스킬의 행동 규칙 명시
- [ ] **모드/워크플로우** 섹션이 존재하는가 — 동작 방식이 분기별로 정의
- [ ] **References 포인터 테이블**이 존재하는가 — 파일명과 역할 매핑
- [ ] **Examples 포인터**가 존재하는가 (해당하는 경우)
- [ ] 섹션 간 탐색이 자연스러운가
- [ ] 각 references 파일 경로가 정확하게 기재되어 있는가

---

## 5. 출력 형식 정의 — 4항목

- [ ] 스킬의 최종 출력 형식이 명확하게 정의되어 있는가
- [ ] 출력 템플릿 또는 예시가 references에 포함되어 있는가
- [ ] 일관된 포맷 규칙이 있는가 (Markdown 테이블, 점수 표기, 등급 표시)
- [ ] 출력 구조가 사용자와 에이전트 모두에게 파싱 가능한가
- [ ] `수행한 검증` 같은 검증 결과 섹션이 출력 계약에 포함되는가

---

## 6. References 품질 — 6항목

- [ ] 관심사 분리가 되어 있는가 — 각 파일이 하나의 주제를 담당
- [ ] 각 파일이 독립적으로 유용한가 — 해당 파일만으로 이해 가능
- [ ] 교차 참조가 정확한가 — 참조된 파일이 실제 존재하고 경로가 맞음
- [ ] 순환 의존성이 없는가 — A→B→A 같은 순환 참조 없음
- [ ] 모든 파일이 완성된 상태인가 — stub이 아닌 실제 내용
- [ ] SKILL.md의 References 테이블에 누락된 파일이 없는가

---

## 7. 자체 평가 메커니즘 — 5항목

- [ ] 스킬이 자기 자신을 진단할 수 있는가 — self-referential 경로 존재
- [ ] 진단 대상에 "Self"가 명시적으로 포함되어 있는가
- [ ] version 필드를 통해 변경 이력을 추적할 수 있는가
- [ ] 품질 기준(metrics)이 정의되어 있는가
- [ ] 이 체크리스트 자체가 스킬 품질의 기준 문서로 기능하는가
- [ ] 구조 검증과 런타임 검증이 별도 스크립트로 분리되어 있는가
- [ ] self 결과가 `logs/self-audit-log.md`에 점수와 변경 메모로 기록되는가

---

## 8. 운영 Guardrail — 5항목

- [ ] CI workflow가 self-audit/doc-lint를 실행하는가
- [ ] PR 템플릿이 self 점수와 검증 명령을 요구하는가
- [ ] Issue 템플릿이 evidence와 proposed change를 요구하는가
- [ ] source of truth와 sync 정책이 문서화되어 있는가
- [ ] maintenance/GC 스캔 경로가 존재하는가

## Self-Maintenance 트리거

이 skill 자체의 drift를 방지하기 위한 재평가 트리거 조건.
아래 상황 발생 시 Self Audit을 재실행하고 Meta 섹션을 갱신한다.

### 재평가 필수 트리거

| 트리거 | 설명 | 영향 범위 |
|--------|------|-----------|
| **파일 추가/삭제** | references/ 또는 examples/ 파일 변경 | SKILL.md 포인터 테이블, 이 체크리스트의 Meta |
| **원칙 변경** | 12원칙의 추가, 삭제, 또는 정의 수정 | principles.md, maturity-framework.md, 체크리스트 |
| **점수 체계 변경** | 점수 기준, 가중치, 계산 수식 변경 | maturity-framework.md, report-template.md |
| **모드 추가** | 새로운 진단 모드 도입 | SKILL.md, workflow, report-template.md |
| **버전 업그레이드** | version 필드 변경 시 | Meta 섹션 진단 시점 갱신 |

### 정기 점검

- **분기 1회**: 전체 skill-checklist 자기 적용 (40항목)
- **사용 후**: 실제 진단 수행 후 워크플로우/템플릿의 현실 적합성 검토
- **외부 변경 시**: Codex skill 시스템의 사양 변경 시 frontmatter/구조 호환성 확인
